Reading an import line

A one-glance cheat-sheet: where does this code actually come from? Four kinds of path, and the @ trap.

Every .ts/.tsx file starts with a stack of import lines. The text after from tells you where that code lives. Four possibilities:

The four kinds

What you see after fromWhere the code livesReal examples in this repo
Bare name
(no ., no @/)		 A third-party package someone installed into node_modules. react, better-auth, drizzle-orm, @tursodatabase/database, expo-router
@rfid/… Our own shared workspace package — the code both apps agree on. @rfid/domain, @rfid/reader-protocol
@/… This app's own src/ folder — an alias (a nickname) defined in tsconfig.json. @/components/Header, @/lib/db, @/lib/utils
./ or ../ A nearby file, by relative path (. = this folder, .. = up one). ./AmendSheet, ../../db/provider
The @ trap (this is what trips everyone) Three things all start with @ but mean different things:
  • @tursodatabase/database — a third-party package (a "scoped" package: @org/name).
  • @rfid/domain — our shared package (also @org/name shape, but the org is us).
  • @/lib/db — NOT a package at all. The slash comes right after the @. This is the "my own src/ folder" alias.
Tell them apart by what follows the @: a word (@rfid, @tursodatabase) = a package; a slash (@/) = the local-src alias.

Where the aliases are defined

These aren't magic — each app declares them once. In apps/web/tsconfig.json:

"paths": { "@/*": ["./src/*"] # "@/anything" means "./src/anything" }

So @/lib/db is exactly apps/web/src/lib/db.ts. Workspace packages like @rfid/domain are wired up a level higher, in the repo's pnpm-workspace.yaml + each package's package.json name.

Worked example — the top of apps/web/src/app/warehouse/page.tsx

# shared brain — used by BOTH apps (types + real functions) import { BUILDING_OPTIONS, inventoryTree, type InventoryType } from "@rfid/domain"; # this app's OWN components (the ui/* ones are shadcn copied into the repo) import { Header } from "@/components/Header"; import { Card } from "@/components/ui/card"; # this app's OWN helpers — @/lib/db is where the real Turso package gets used import { getDb } from "@/lib/db"; import { cn } from "@/lib/utils";

Nothing on that page imports a third-party package directly. The only external dependency (Turso) arrives through @/lib/db, which is the team's own file that imports the real @tursodatabase/database package by its bare name and wraps it. That "wrap external tools in your own lib/ file" habit is a common, healthy pattern.

One more distinction import { type InventoryType } — the type keyword marks a type-only import: a data-shape description used while writing/checking code that disappears when the app runs. Without type (e.g. inventoryTree, BUILDING_OPTIONS), you're importing real code that executes.
