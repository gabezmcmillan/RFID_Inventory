# Starting point: beginner, two-week horizon, wants to reach "small changes"

The learner has little to no prior web-development experience and is coming to this
codebase cold. They have a **~two-week window** to become able to maintain and make
changes to it, and they're explicitly using the project as a way to learn general
web-dev practices — so lessons should teach the transferable idea alongside the
repo-specific fact.

Stated depth target: concepts + the map, but one level deeper on the *important*
parts — enough to **read and reason about** key code and make **small, safe
changes**, not to build large features unaided (yet).

Implications for teaching:
- Prioritize the code paths most likely to need maintenance in two weeks over
  breadth/theory. Prefer "follow this real file" over abstract explainers.
- Do not assume any web vocabulary; keep defining jargon on first use.
- Coverage in lesson 1 (client/server/DB, framework, monorepo, TypeScript, React,
  Expo, Next.js, Turso, ORM) is exposure only — none proven yet. Watch for evidence
  before promoting any to GLOSSARY.md.
