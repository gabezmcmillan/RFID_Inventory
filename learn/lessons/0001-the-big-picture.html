RFID Inventory · Lesson 1

The Big Picture: what got re-written, and why

A map of the whole codebase for someone who has never done web development — and how the new version differs from the old one.

Before we touch a single line of code, you need a mental map. Once you can picture where things live and why, every later lesson has somewhere to hang. That is the only goal of this lesson: by the end you should be able to open the repo and say, out loud, what each folder is for.

Analogy Think of the whole system as a warehouse and a storefront. In the back, staff receive boxes and put them on shelves. Out front, customers browse what's in stock and ask for things. The re-write didn't change that story — it changed the tools the staff and customers use, and the ledger everyone writes in.

1. What does this system actually do?

It tracks physical inventory using RFID tags — little chips stuck to boxes that a handheld reader can detect by radio. There are two kinds of people using it:

  • Warehouse staff — receive truckloads, scan each box's tag to check it in, later scan again to check it out when it ships to a jobsite. They can also sweep a room to count everything, or play "hotter/colder" to physically find one specific tag.
  • Jobsite / office people — can't touch the warehouse computer. They open a website to see what's in stock and submit a material request ("please send me 4 of these"). The warehouse sees the request and fulfils it.

Both the old and new versions do exactly this. Keep that in mind — the job is unchanged, only the machinery was rebuilt.

2. A few words you'll need first

You said you're new to web dev, so here are the load-bearing terms. Don't memorize — just get the gist; they'll recur.

  • Client — the thing a person actually touches (a phone app, or a web page in a browser). Also called the front-end.
  • Server — a program running on some computer elsewhere that the client talks to over the internet. Also called the back-end.
  • Database — the organized, permanent memory where all the facts live (every box, every request). The "ledger" in our analogy.
  • Framework — a big pre-built toolkit that gives an app its skeleton so you don't rebuild the basics. (Next.js and Expo below are frameworks.)
  • Package / monorepo — a "package" is a self-contained chunk of code with a name. A monorepo ("mono" = one) is a single repository holding several apps and shared packages together, so they can share code. This repo is a monorepo.

3. The old system (Python)

The version being replaced — still documented in the repo's README.md and living under apps/warehouse/ and apps/cloud/ — was written in the Python language and came in two separate programs that barely shared any code:

Old · warehouse

apps/warehouse/

  • A Python program you install on a Windows PC in the warehouse (bundled into one .exe file).
  • The staff used it through a web browser pointed at that same PC (127.0.0.1:8000).
  • Talked to the RFID reader over a USB cable, and scanned bills-of-lading on an Epson desktop scanner.
  • Saved everything to a SQLite file (inventory.db) sitting on that one PC.
Old · cloud

apps/cloud/

  • A second, separate Python program hosted on the internet (on Vercel).
  • Showed jobsite users a read-only stock list and a request form.
  • Had its own separate database (Postgres) — a copy of the warehouse data.
The hard part of the old design Two programs, two databases, in two places. Someone had to write code by hand to constantly copy data from the warehouse PC up to the cloud and pull requests back down. That copying machinery — apps/warehouse/sync.py plus a packages/contract defining exactly which columns to copy — was fiddly, easy to break, and a lot of the total code. Remember this pain; the re-write exists largely to erase it.

4. The new system (TypeScript)

The re-write is written in TypeScript (a stricter, typo-catching cousin of JavaScript — the language of the web). Instead of two isolated Python programs, it's four cooperating pieces in one monorepo, three of which you already have open a lot:

apps/field/ # the iPhone app — replaces the warehouse PC entirely apps/web/ # the new website — jobsite stock, requests, office browse packages/domain/ # shared brain: the data shape + all the business rules packages/reader-protocol/ # how to talk to the RFID reader hardware

apps/field — the iPhone app

Built with Expo, which is the standard toolkit for React Native — a way to build real native phone apps using the same React ideas the web uses. The phone clips onto the RFID sled and talks to the reader over Bluetooth (no more USB cable, no more Windows PC). The app/ folder inside it has one file per screen — check-in.tsx, check-out.tsx, sweep.tsx, finder.tsx, warehouse.tsx, requests.tsx — which line up almost exactly with the "Modes" listed in the old README. Bills-of-lading are now photographed with the phone camera instead of a desk scanner.

apps/web — the new website

Built with Next.js, the most common React framework for websites, also hosted on Vercel. It does what the old apps/cloud/ did (stock browsing, a request cart, order status) plus an office-facing warehouse view, and it now has real sign-in (Microsoft company login, via a library called Better Auth). Screens live under src/app/ (warehouse/, cart/, requests/, tag/, sign-in/).

packages/domain — the shared brain

This is the clever part and the biggest change. All the rules and the shape of the data — what a "tag" is, what a "request" is, how check-in works — live once here, in plain TypeScript. Both the phone app and the website import this same package. In the old world those rules were duplicated across the two Python programs; now there's a single source of truth. It's a direct port of the old apps/warehouse/config.py (constants) and db.py (database logic).

The database: one Turso, not two

The new system uses a single Turso database (Turso is a modern, cloud-hosted version of that same SQLite technology). Here's the magic: the phone keeps a local copy it can read and write even with no signal, and Turso automatically syncs that copy with the cloud when there's a connection. The website reads and writes the very same cloud database directly. Because Turso handles the copying, the entire hand-written sync machinery from the old system (sync.py, packages/contract) simply gets deleted.

The one big idea The re-write trades two Python programs + two databases + hand-written copying for two TypeScript apps + one shared brain + one auto-syncing database. Almost every other difference flows from that single decision.
