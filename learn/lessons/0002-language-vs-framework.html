Language vs. library vs. framework

Untangling TypeScript, React, React Native, Next.js and Expo — and why apps/web and apps/field are more alike than they look.

You asked exactly the right question: "is React Native a framework for writing TypeScript?" Almost — but the one word to fix is what unlocks everything. So let's fix it first.

The misconception to drop React, React Native, Next.js and Expo are not "frameworks for writing TypeScript." Only TypeScript is a language. The others are tools you use while writing TypeScript — and they stack on top of one another. Sorting out what layer each one lives on is the whole lesson.

1. The four things, one sentence each

  • TypeScript — the language you type. It's JavaScript (the language of the web) plus a "type system" that catches typos and mismatched data before the app runs. This is the actual text in every .ts / .tsx file.
  • React — a library for building a user interface out of reusable components (a button, a form, a list row). React is just the idea and the rules; on its own it doesn't know whether it's drawing to a phone or a browser.
  • React Native / React DOM — the two renderers that give React somewhere to actually draw. React DOM draws into a web browser; React Native draws to a real iOS/Android screen. Same React, different destination.
  • Next.js / Expo — the frameworks that wrap everything above into a productive whole: routing between screens, a dev server, the build, connections to device features. Next.js wraps React-for-web; Expo wraps React Native.
Analogy — writing and publishing a book TypeScript is the language you write in (English). React is a style guide — "compose everything out of small, reusable sections." React DOM vs React Native is the format you publish in — a printed book vs an audiobook: same words, different medium. Next.js / Expo is the publishing house that handles binding, printing, and distribution so you only worry about the writing.

3. Why layer at all? (the transferable idea)

This "stack" pattern is everywhere in web development, so it's worth internalizing beyond this repo. Each layer exists so the layer above doesn't have to re-solve a hard problem:

  • You could write a phone app in raw TypeScript talking directly to iOS — but you'd hand-build every button and screen transition.
  • React spares you that by letting you describe UI as components.
  • But React alone doesn't handle navigation, building an installable app, or camera access.
  • Expo spares you that. You get to spend your time on the warehouse logic, not plumbing.

The trade: each layer you adopt is power you don't have to write, in exchange for learning that layer's conventions. That's the deal web frameworks offer.

4. See it in this repo

Because React is shared, a "screen" in each app looks strikingly similar — both are TypeScript files ending in .tsx that return a component:

  • Field (Expo): screens live in apps/field/app/ — e.g. check-in.tsx, warehouse.tsx. The file name is the route; that's an Expo feature (expo-router).
  • Web (Next.js): screens live in apps/web/src/app/ — e.g. warehouse/, cart/, requests/. The folder name is the route; that's a Next.js feature (the "App Router").

So when you open a .tsx file: the syntax is TypeScript, the components and useState/useEffect are React, the on-screen elements (<View> for the phone vs <div> for the web) tell you which renderer, and the routing/config conventions are Expo or Next.js.

Handy tell Which renderer am I looking at? If you see <div>, <button>, <p> — that's React DOM (web). If you see <View>, <Text>, <Pressable> — that's React Native (phone). Phones have no "div."

5. What you should now be able to say

  • "TypeScript is the language; the other three are libraries/frameworks I use while writing it."
  • "React is the UI idea; React DOM and React Native are two renderers for it."
  • "Next.js wraps React-for-web; Expo wraps React Native. Both apps share the bottom two layers."
